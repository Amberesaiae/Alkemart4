## Short answer

**Not “all e2e fully baked.”** Core marketplace spine (RBAC, isolation, catalog, ATC/COD shell, surface smoke) is automated. Ghana quick-setup UI, Paystack money path, browser full checkout, and empty-shop browser create remain **partial or manual**.

---

## Automated suites

| Command | What it proves | Bake-in |
|---------|----------------|---------|
| `bun run smoke:unit:api` | Pure unit: isolation middleware, catalog-map, catalog-cache keys/TTL, quality, sellable, gates | Always run on API change |
| `bun run smoke:onboarding` | Health, unauth 401s, optional authed onboarding/moderation | Fast preflight |
| `bun run smoke:catalog` | Playwright: P1.1 strategy, offer_id, min_price; P1.4 cache hit after warm | Catalog contract |
| `bun run smoke:ui` | Surfaces: shop brand, admin/seller login, language control | Layer 3 UI |
| `bun run smoke:rbac` | Playwright serial: API deny/allow, product isolation, multi-store UI, admin ops, cross-role, catalog+PDP, product create+image, moderation | Multi-vendor RBAC |
| `bun run smoke:live` | Shell human flow: catalog → vendor login → isolation → images → product → offer → admin publish → ATC offer_id → COD → lookup | Deep Mode B spine |
| `bash scripts/ensure-p13-marketplace-indexes.sh` | Idempotent DB indexes (P1.3) | Ops, not app test |
| `bun run smoke:all` | onboarding + catalog + ui + rbac | Default gate |
| `bun run smoke:all:deep` | unit:api + live + smoke:all | Release/demo |

## Architecture rules covered by tests

| Rule | Where proven |
|------|----------------|
| ATC = `offer_id` only | live-e2e ATC step; catalog asserts `offer_id` |
| Seller list = owned only | RBAC phase 1; live-e2e isolation; unit middleware |
| Catalog = published + offer + open seller | live catalog; Playwright phase 5 |
| No Mercur shared-catalog leak | LAB_LEAK title filter + middleware |
| Catalog light → heavy page | `strategy: light_ids_then_heavy_page` in live + Playwright |
| Catalog Redis cache | live-e2e 1b; Playwright catalog-performance |
| Indexes | ensure-p13 SQL applied + verified on Neon |

## Definition of “well baked”

A change is **baked** when:

1. Unit covers pure logic (if any).  
2. Live HTTP or Playwright asserts the **product rule** (not only “HTTP 200”).  
3. This matrix + performance/isolation docs name the suite.  
4. Root `package.json` exposes a `smoke:*` entry.
