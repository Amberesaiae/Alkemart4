# Alkemart Full-Stack Audit — Architecture, Accessibility & Ghana/Africa Fit

**Scope:** `artifacts/api-server` (Express + Drizzle + CASL), `artifacts/alkemart` (React + TanStack Router/Query + shadcn/ui), `lib/db`, `lib/abilities`, `lib/api-spec/openapi.yaml`.
**Method:** static code review (routes, schema, checkout transaction, ability rules, frontend routes/components), no live user testing. This document is analysis only — no application code was changed.

---

## 1. Architecture end-to-end

### Request flow
Frontend (TanStack Router + React Query hooks generated by Orval from `lib/api-spec/openapi.yaml`) → Express routes (`artifacts/api-server/src/routes/*`) → `authSessionMiddleware` (attaches `req.user` + `req.ability` from a cookie session token) → `requireAbility(action, subject)` guard (CASL, `lib/abilities`) → Drizzle ORM queries against Postgres (`lib/db/src/schema`).

This is a genuinely sound, first-principles-consistent design for its scale:
- **OpenAPI-first contract**: the spec is the single source of truth for both server validation (Zod, generated into `@workspace/api-zod`) and the typed React Query client (`@workspace/api-client-react`). Frontend and backend cannot silently drift on shape.
- **Checkout correctness**: `runCheckoutWorkflow` (`artifacts/api-server/src/lib/checkout.ts`) is a single Postgres transaction that reserves stock with a conditional `UPDATE ... WHERE stock - reservedStock >= qty`, which is the correct way to prevent overselling under concurrent checkouts (relies on Postgres row locking, not an app-level read-then-write race). Order totals, promo discounts, and line-item price/title are all snapshotted at purchase time rather than re-derived later — correct for historical accuracy.
- **RBAC**: CASL abilities are defined once (`lib/abilities`) and shared between server route guards and (presumably) frontend conditional rendering, avoiding duplicated authorization logic. Per-role vendor scoping is computed correctly per role assignment (not pooled across roles).
- **Append-only payment ledger**: `order_payment_events` is modeled Medusa-style (append events, don't mutate a single status column), which is the right shape for eventually wiring a real gateway (auth → capture → refund → dispute) without a schema rewrite.

### First-principles gaps / architecture findings

| Severity | Finding | Detail |
|---|---|---|
| **P0** | No real payment capture | Checkout always inserts a synthetic `paid` event (`checkout.ts:143-152`) regardless of any actual money movement. The `vendors.paystackRecipientCode` column exists but nothing writes to it or calls Paystack. This means **every order is "paid" whether or not the buyer paid**, which is a critical business-logic gap, not just a missing integration. |
| **P0** | Disputes are not linked to real orders | `disputes.orderRef` is a freeform text field (`lib/db/src/schema/disputes.ts:8-13`), predating the orders table. Admin/support cannot join a dispute to actual order/fulfillment/payment data — disputes are effectively unstructured tickets, undermining any resolution workflow. |
| **P1** | No inventory-aware read path | `GET /products` and `GET /products/:id` return `stock`/`reservedStock` but the frontend has no visible "N left" / out-of-stock badge logic audited here beyond `isActive`; a product can show as purchasable in listings up to the instant of checkout, with the only failure surface being a 409 at final submission. |
| **P1** | Search is substring-only, no ranking | `catalog.ts:54-58` uses `ILIKE '%term%'` OR'd across title/brand with no ranking, typo tolerance, or full-text index — will not scale past a small catalog and won't match local product-naming conventions (transliterations, brand abbreviations). |
| **P1** | No cancellation/refund transition | `ORDER_STATUSES` includes `cancelled` but no route mutates an order into it; `FULFILLMENT_STATUS_TRANSITIONS` is forward-only (`unfulfilled → packed → shipped → delivered`) with no reversal path for a failed delivery, wrong item, or buyer-initiated cancellation before shipment. |
| **P2** | No reviews/ratings write path | `products.ratingAvgX100` / `ratingCount` and `vendors.ratingAvgX100` / `ratingCount` exist as columns but there is no route to submit a review — they can only ever read as zero. |
| **P2** | Offset pagination on products | `LIMIT/OFFSET` (`catalog.ts:66-70`) plus a separate COUNT query on every list call; fine at current scale, will degrade and can skip/duplicate rows under concurrent writes as the catalog grows. |
| **P2** | Commission model undocumented in flow | `vendors.commissionBps` exists but no code path (payout calculation, admin report) reads it — vendor payouts are entirely unmodeled beyond the column. |

## 2. Accessibility (WCAG 2.1 AA) & cross-device

### What's working
- The UI is built almost entirely on Radix UI primitives via shadcn/ui (`src/components/ui`), which provides correct-by-default keyboard navigation, focus trapping (Dialog/Sheet), and ARIA roles for complex widgets (Accordion, Tabs, Dropdown, Select).
- Semantic landmarks are used (`<nav>`, heading hierarchy, `<aside>` for checkout summary).
- Mobile-first Tailwind breakpoints throughout, plus a dedicated `use-mobile` hook for JS-level responsive branching.
- Color tokens (`index.css`) use a high-contrast ink-on-off-white primary palette (`oklch(0.15 0 0)` text on `oklch(0.965 0 0)` background), which comfortably clears WCAG AA contrast; the goldenrod accent (`oklch(0.76 0.145 82)`) is paired with dark ink text, not light text, which is also contrast-safe.
- Loading states use skeletons rather than layout-shifting spinners; empty states are explicit (e.g. "Your cart is empty").

### Findings

| Severity | Finding | Detail |
|---|---|---|
| **Serious** | Plain `<button>` elements without accessible labeling in custom cards | `AddressCard` and `PaymentMethodCard` (`src/components/shop/address-card.tsx:36-37`, `payment-method-card.tsx:38-39`) render bare `<button>Edit</button>` / `<button>Remove</button>` with no `type="button"` and no context (e.g. "Edit Ama Boateng's address") — ambiguous for screen reader users navigating by control list, and missing `type` risks accidental form submission if ever nested in a `<form>`. |
| **Serious** | Icon-only controls: verify all have `aria-label` | Multiple `size="icon"` buttons exist across the app (close, remove-from-cart, wishlist heart, carousel arrows, etc.); `aria-label` usage exists in several components but was not exhaustively cross-checked against every icon-only control in this pass. Every icon-only button needs an explicit accessible name — this needs a full, tool-assisted pass (see recommendation below), not spot-checking, since Radix's own primitives (e.g. `DialogClose`) don't auto-label custom icon children. |
| **Moderate** | Only one raw `<img>` tag was found project-wide (rest go through `ImageSlot`) | Good pattern in principle — confirm `ImageSlot` always requires and forwards a meaningful `alt` (it does in the checkout line-item review, `alt={line.product.title}`), and audit that decorative-only images/icons consistently use `aria-hidden="true"` (seen correctly in `SnapEbtBadge`'s icon, `snap-ebt-badge.tsx:11`) rather than being left in the accessibility tree. |
| **Moderate** | No visible focus-order audit for the checkout accordion | The checkout page (`_shop.checkout.tsx`) nests four accordion steps where step 1–3 render **static, non-interactive mock content** (hardcoded address/payment cards with disabled-looking "coming soon" copy) inside an otherwise real form flow. A screen-reader or keyboard user has no way to distinguish "this control does nothing yet" from "this control is broken," which is a usability/trust problem as much as an accessibility one. |
| **Moderate** | Low-end device / low-bandwidth tolerance not verified | No evidence of image lazy-loading conventions, responsive `srcset`, or bundle-size budgeting audited in this pass. Africa's mobile-heavy, often 3G/data-constrained user base makes this materially more important than for a typical Western SPA audience; needs a dedicated network-throttled pass (Lighthouse mobile + 3G throttling) before launch. |
| **Minor** | No formal i18n framework | No `react-i18next`/`formatjs` or `t()` calls found anywhere in the frontend; all copy is hardcoded English strings embedded directly in JSX. Not a WCAG blocker per se, but it forecloses future language support (Twi, Ga, Ewe, Hausa, French for regional/ECOWAS expansion) without a full rewrite of every component. |

**Recommendation:** run an automated pass (axe-core or Lighthouse accessibility audit) against the live preview once seed data exists, to catch contrast/labeling issues this static read cannot (e.g. computed contrast of muted-foreground text at small sizes, dynamic ARIA state on custom components). This audit intentionally stayed in-scope of a manual/code-based review; tooling is a fast, cheap follow-up.

## 3. Ghana/Africa fit gaps

This is the most consequential section: the product was scaffolded from a template built for a different market, and several artifacts of that origin remain even after a partial "Ghana-ization" pass.

| Severity | Finding | Detail |
|---|---|---|
| **P0** | No real mobile money integration | The UI *labels* things as "MoMo" (`SnapEbtBadge` renders "MoMo eligible"; `PaymentMethodCard` defaults to "MTN Mobile Money") but there is no MTN MoMo, Vodafone Cash, Telecel Cash, or AirtelTigo Money API integration anywhere in `api-server`. No Paystack, Flutterwave, or Hubtel SDK/webhook code exists despite the `paystackRecipientCode` column on `vendors`. In a market where mobile money is the dominant payment rail (far more than cards), this is the single highest-priority gap. |
| **P0** | Leftover US-benefits-program component (`SnapEbtBadge`) | `src/components/shop/snap-ebt-badge.tsx` is named and shaped after the **US SNAP/EBT** government food-assistance payment program — a payment concept with zero relevance in Ghana. It's been repurposed to say "MoMo eligible" but the component identity, and the underlying assumption that only *some* products are payment-method-eligible (a rule specific to US SNAP restrictions), doesn't map to any real Ghanaian program and should be removed or fully re-conceived rather than relabeled. |
| **P0** | Checkout defaults to "pending payment" / pay-on-delivery without a real COD workflow | The checkout code comment says the recorded payment event is "pay-on-delivery style default" (`checkout.ts:151`), but there's no actual cash-on-delivery flow: no COD confirmation step, no rider/agent cash reconciliation, no COD-vs-prepaid order distinction visible to admin/vendor. If COD is the intended fallback (common and often necessary in Ghana/West Africa for trust reasons), it needs to be a first-class, explicit payment method — not an implicit default disguised as "paid." |
| **P0** | Address model has no real data — entirely hardcoded/mocked | `_shop.account.addresses.tsx` renders a **static, hardcoded array** of three sample Ghanaian addresses (`addresses.tsx:19-23`) — there is no backend table, no CRUD API, and the "Add address"/"Edit"/"Remove" buttons don't wire to anything. Checkout's address step explicitly says "Address management is coming soon" (`_shop.checkout.tsx:83-85`) and always ships to a fabricated default. This means **the app cannot actually collect a real delivery address today** — a hard blocker for any real order. |
| **P1** | No Ghana Post GPS / digital address support | Given there's no address system at all yet, this is a design decision to get right the first time: Ghana's official addressing system (GhanaPostGPS digital codes, e.g. `GA-183-8341`) plus landmark-based description fields (common necessity given inconsistent street addressing) should be first-class fields, not a bolted-on afterthought to a US-style street/city/zip form. |
| **P1** | No local logistics/courier integration | `fulfillments` is explicitly scoped as "internal shipment/delivery status tracker only (no external carrier integration)" (`fulfillments.ts:8-9`). There's no dispatch-rider assignment, no integration with local last-mile providers, and delivery status changes are entirely vendor-self-reported with no buyer-facing tracking beyond a status label. |
| **P1** | Currency handling is GHS-correct but tax/VAT is entirely absent | Money is correctly modeled in pesewas (integer minor units, avoiding float rounding bugs) and displayed with `GH₵`, which is good practice. However there is no VAT/NHIL/GETFund levy calculation anywhere in the order total — Ghanaian e-commerce transactions are subject to VAT and associated levies, and silently omitting them is both a compliance gap and a source of buyer confusion at any future reconciliation. |
| **P2** | No SMS notification channel | `notifications` table/UI is in-app only; no SMS fallback exists despite SMS being significantly more reliable than app push/email for reaching Ghanaian buyers and vendors on entry-level phones or with intermittent data. |
| **P2** | No offline/low-connectivity resilience strategy | No service worker, no optimistic UI beyond React Query defaults, no explicit "you're offline" handling audited — same underlying concern as the accessibility section's bandwidth finding, but worth calling out as a market-fit issue: Africa's mobile networks have materially different reliability than assumed by a typical SPA scaffold. |
| **P2** | English-only content in a multilingual market | Twi, Ga, Ewe, Hausa, and French (for cross-border ECOWAS commerce) are all realistic language needs with zero groundwork (no i18n framework, see accessibility section) laid for them. |

## 4. End-to-end workflow & gap audit

### Guest browsing → discovery
- **Works:** category/tag/vendor/search filtering, homepage sections config (admin-editable), image moderation pipeline before anything renders publicly.
- **Gaps:** search is substring-only (see §1); no faceted filtering (price range, rating, in-stock only); no sort parameter despite `ProductList` shape suggesting it could exist; no "recently viewed" or recommendation surface; no explicit handling audited for zero-result search states beyond an empty list.

### Signup / login
- **Works:** email+password signup/login, session cookie, `/auth/me`.
- **Gaps:** no phone-number-based signup/login despite `users.phone` existing as a column — in a market where phone number is often the primary identifier (and tied to mobile money), email-only auth is a mismatch. No password reset flow found. No OTP/SMS verification. No social/OAuth login.

### Product detail page (PDP)
- **Works:** vendor info, images via moderation pipeline.
- **Gaps:** no reviews display (no write path exists either, see §1); no related/similar products audited; no explicit stock-count or "only N left" urgency messaging tied to the real `stock - reservedStock` value.

### Cart
- **Works:** session-based (works for guests) via `cartSessionKey`, add/update/remove, subtotal computed server-side.
- **Gaps:** no visible stock-check on cart view (a stale cart with an item that's since sold out only surfaces the problem at final checkout, as a generic 409); no saved-for-later / move-to-wishlist-from-cart flow audited; no explicit multi-vendor cart grouping presentation (an order can span vendors — fine on the backend, but the buyer-facing cart should communicate "these ship separately").

### Checkout & payment
- **Works:** transactional stock reservation, promo code validation (active window, min order, usage limit — all correctly race-safe inside the same transaction).
- **Gaps (critical):** address step is non-functional (mocked data only); payment step is non-functional (hardcoded card/MoMo display, no real charge); the only real "payment" outcome is a synthetic always-succeeds event. **This means the entire checkout flow currently cannot fail for a real business reason (declined payment, invalid address) — it can only fail on empty cart, stock, or promo code.** That's a much narrower failure surface than production checkout requires. No guest checkout path was found — `beforeLoad: requireAuthBeforeLoad` forces login before reaching checkout, which is a legitimate design choice but should be a stated decision, not a byproduct of unfinished address/payment work.

### Order tracking / post-purchase
- **Works:** buyer order list/detail, vendor order list scoped to their own vendor(s), forward-only fulfillment status transitions with timestamps per stage, in-app notifications on `order.placed`.
- **Gaps:** no buyer-facing cancellation before a vendor packs; no refund path even though `order_payment_events` supports a `refunded` type; no re-order/"buy again" action audited; no delivery ETA or rider tracking beyond a status label; ratings/reviews cannot be left post-delivery (no write path).

### Returns / disputes
- **Works:** a disputes queue exists for admin/support with status tracking (`open`/`resolved_buyer`/`resolved_seller`).
- **Gaps (critical):** disputes aren't linked to real orders (freeform `orderRef` text, see §1) — so a support agent resolving a dispute has no structured view of what was ordered, paid, or fulfilled. No buyer-initiated "start a return/dispute" entry point was found wired to an actual order (only the admin-side queue view was confirmed). No refund execution — dispute resolution is view/status-only by explicit design comment (`updateAdminDispute` summary: "no financial actions").

### Vendor onboarding & management
- **Works:** vendor status (active/suspended) with admin control, scoped product CRUD via CASL vendor-ownership conditions, vendor order/fulfillment views correctly scoped by vendor ID.
- **Gaps:** no self-serve vendor signup/application flow (per existing project memory, vendor roles must be DB-inserted manually) — every vendor onboarding today is a manual admin action, which won't scale past a handful of vendors. No vendor payout view/report despite `commissionBps` existing. No vendor-side analytics (sales, top products).

### Admin moderation
- **Works:** vendor approval/suspension, image moderation (approve/reject with reason), homepage section configuration, promotions CRUD, dispute status updates, admin-initiated conversations.
- **Gaps:** no audit log of admin actions (who suspended a vendor, when, why) — `reviewedByUserId`/`reviewedAt` exist on images but not consistently across other admin-mutated entities (vendor status, dispute resolution). No admin-side order search/filtering beyond a flat newest-first list (`GET /admin/orders`) — will not scale as order volume grows.

### Messaging / notifications
- **Works:** buyer/vendor-initiated support conversations, admin can open conversations with any user, in-app notification list with read/unread state.
- **Gaps:** notification types are a fixed enum of exactly two values (`order.confirmed`, `order.new_for_vendor`) — no dispute-update, fulfillment-status-change, or promotion notifications despite those being natural triggers already emitted as domain events (`fulfillment.status_changed` is emitted in `checkout`/`orders` but nothing appears to subscribe to it for a notification). No email/SMS delivery channel (see §3).

## 5. Prioritized recommendations

### P0 — blockers for a real, trustworthy transaction
1. **Build a real address system**: `addresses` table (Ghana Post GPS digital code + landmark description + city/region + phone), CRUD API, and wire the checkout address step to it instead of the hardcoded mock.
2. **Integrate a real payment path**: at minimum Paystack (which already supports Ghanaian mobile money rails) or a direct MTN MoMo/Hubtel integration, replacing the synthetic always-`paid` event; make cash-on-delivery an explicit, first-class payment method if it's meant to remain supported, not an implicit fallback.
3. **Remove/replace the `SnapEbtBadge` component** and its "eligibility" concept — it encodes a US government program with no Ghanaian equivalent; if a "pay with MoMo" indicator is wanted, build it as a clearly-named, purpose-built component.
4. **Link disputes to real orders** (proper `orderId` FK, replacing freeform `orderRef`) so support agents can act on structured data.

### P1 — needed before scaling past a pilot
5. Add VAT/levy calculation to order totals (compliance + trust).
6. Add order cancellation and a refund execution path (`order_payment_events` already supports `refunded`; nothing writes it yet).
7. Replace substring search with a proper full-text/trigram search and add sort/facet parameters.
8. Add phone-number-based signup/login and password reset.
9. Build self-serve vendor application/onboarding instead of manual DB role grants.
10. Full accessibility pass with automated tooling (axe-core/Lighthouse) against seeded data, focused on icon-only button labeling and the checkout accordion's "coming soon" static content.

### P2 — strengthens Africa/Ghana market fit and long-term scale
11. Introduce an i18n framework (even English-only today) so Twi/Ga/Ewe/Hausa/French can be added without a rewrite.
12. Add SMS notification delivery alongside in-app notifications.
13. Add a reviews/ratings write path (columns already exist, unused).
14. Add local courier/dispatch-rider integration or at least a structured manual dispatch assignment flow.
15. Add admin audit logging across all moderation actions, not just images.
16. Move product listing from offset pagination to keyset/cursor pagination ahead of catalog growth.
